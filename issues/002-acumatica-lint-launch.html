ISSUE 002 · ACUOPS LAUNCH

60 ways an Acumatica publish silently fails. And one Python file that catches them.

We open-sourced the static-check engine inside AcuOps. Free, Apache 2.0, pip install acumatica-lint today.

BY KEVIN BIBELHAUSEN · 2026-05-21

A DAC extension shipped to production without a matching <Sql> ALTER TABLE. The publish reported success. The field appeared in $adHocSchema. PUTs returned HTTP 200 with the value echoed in the response body.

The value was silently dropped on commit. Every write. For seven days.

We found it because a customer noticed a spreadsheet didn't match the order screen.

That kind of failure mode is what we built acumatica-lint to catch. Today it's free, open source, Apache 2.0, and on PyPI: pip install acumatica-lint.

About 60 specific things can go wrong when Acumatica publishes a customization, and most of them produce HTTP 200 from somewhere on the way down. The publish log says "Validates Sql Scripts" without telling you whether any scripts ran. The GI catalog excludes a screen without raising an error. The DAC extension's value lands in $adHocSchema whether or not the endpoint actually maps it.

None of these are documented in one place. None of them are in Acumatica's official analyzer (Acuminator). Each one costs about 14 hours when it surfaces on a production tenant during business hours.

We've been running Acumatica deployments at scale for Ästhetik — a mid-market textile group, and the platform company at the heart of Studio b.'s operating portfolio — for years. Every incident produced a rule. The rules turned into validate-project.py. Today that file is acumatica-lint.

Twenty-five hard-fail validators. Every one encodes a real production incident.

Here is what it catches. Five examples; the full list is in the repo.

DAC fields without matching <Sql> ALTER TABLE. Acumatica's auto-column creation has been broken on cloud since early 2026. A [PXDBString] field with no corresponding <Sql> ALTER will fail at publish — silently, with no error in the publishEnd log. The customization "deploys." The field doesn't exist.

GI Creation Contract. Every new OData-exposed Generic Inquiry requires <RolesInGraph Rolename="*" Accessrights="4" ApplicationName="/" /> in its SiteMap row AND a matching entry in the package's CustomizationPlugin allowlist. Miss either and the GI catalog-excludes, returning HTTP 404 from /OData/{tenant}/$metadata. No error. The publish reports success.

SiteMap orphans. A row with ParentID="00000000-0000-0000-0000-000000000000" is invisible in SM205010 (Access Rights by Role), ungrantable, and returns OData 404. The publish doesn't flag it because the row exists — just unreachable.

Modern GI format (24.208+). <GITable Name> must be the fully-qualified DAC name — PX.Objects.PO.POOrder, not bare POOrder. <GIWhere Condition> uses "E " / "NE" / "GE" — not "EQ". The attribute is Operation=, not Operator=. Each is a silent catalog exclusion at publish time.

<Sql> re-execution. Acumatica tracks scripts by Name. Bumping a script to _v2 is silently skipped — the script doesn't appear in publishEnd at all. For logic that must survive every publish, use CustomizationPlugin.PXDatabase.Execute(), not <Sql>.

The static checks aren't where the operational moat is.

The hard part is continuous gating — running the checks on every PR, in a sandbox dry-run before prod, with snapshot/rollback that actually rolls back when something fails midway, AI review on the C# code that static analysis can't catch, Slack alerts when a publish breaks, and enforcement of deploy-window rules so schema changes don't ship at 11am on a Tuesday.

That's AcuOps. $800 per Acumatica instance per month. The linter is the engine; AcuOps is the pipeline that runs the engine fifty times a day without you thinking about it.

The linter tells you what's wrong. AcuOps prevents broken from shipping.

We open-sourced the linter for two reasons. First: VARs and in-house teams can run the same checks Studio b. runs against its own deployments, even if they never become AcuOps customers. That's value we'd rather make free than gate. Second: the rules need to spread. Every Acumatica deployment that hits one of these failure modes for the first time pays the 14-hour tax. We'd rather encode the lesson once and ship it everywhere.

Three Roslyn analyzer rules are also queued upstream to Acumatica's own Acuminator project.

The first one — PX1121 (MissingSchemaMutationGuard) — flags CustomizationPlugin.UpdateDatabase() schema mutations that lack IF NOT EXISTS / IF EXISTS guards. It's PR #671 against Acumatica/Acuminator:dev. Two more analyzers are authored and waiting for PR #671's review feedback before filing in sequence.

These cover patterns Acuminator's existing 39 diagnostics don't catch. If Acumatica accepts them, every Acumatica customization developer using Visual Studio benefits — not just AcuOps customers.
